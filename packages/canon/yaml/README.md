# Canon — L2 Canonical Vocabulary

This directory is the **authoritative** controlled vocabulary for L2 canonicalization. Every agent profile in the database is tagged against codes defined here. The files in this directory — not the database, not the code — are the source of truth.

## Files

| File | What it is |
|---|---|
| `VERSION` | Current canon version. Bump on every change. |
| `thema_subjects.yaml` | Thema subject codes (fiction F*, children's Y*, memoir/poetry/plays D*). Generated from EDItEUR v1.6 — **do not edit by hand**. |
| `thema_audience.yaml` | Thema 5A* audience-age qualifiers. Generated — do not edit by hand. |
| `thema_form.yaml` | Thema X* form qualifiers (graphic novels / manga). Generated — do not edit by hand. |
| `extensions.yaml` | Local non-Thema additions (`LOCAL:*` prefixed). Hand-edited. |
| `hard_nos.yaml` | Hard-no tags (not part of Thema). Hand-edited. |
| `aliases.yaml` | Raw phrase → canonical code. Hand-edited. The runtime L2 consults this. |
| `coverage_report.md` | Most recent dry-run validation output. Regenerated by `scripts/canon_coverage.py`. |
| `_source/` | Upstream Thema JSON and the one-shot filter script. Audit trail only. |

## Versioning

Semver-ish:

- `v0.x` — pre-Step-4. Canon is seeded from Thema + domain knowledge, validated only against the 7 benchmark profiles. **Unstable.**
- `v1.0` — post-Step-4 lock. Canon has been validated against ≥200 real profiles with coverage and κ metrics reported. **Stable.** Downstream layers (L3 embeddings, Step 6 matcher) can assume this is frozen.
- `v1.x` — controlled additions during review cycle (see below).
- `v2.0` — next Thema upstream version bump or major structural change.

Every tagged profile in the database stores the canon version that tagged it. When the canon changes, the Canonicalizer decides whether to re-tag based on the diff.

## Facets

L2 produces five facets per profile, each backed by an orthogonal codebook. They correspond to the sections of `aliases.yaml`.

| Facet | Source of codes | Used by matcher for |
|---|---|---|
| subject | `thema_subjects.yaml` + `LOCAL:*` in `extensions.yaml` | Genre match (primary + sub-tag overlap) |
| audience | `thema_audience.yaml` + `LOCAL:*` in `extensions.yaml` | Audience-proximity score (ordered by age) |
| form | `thema_form.yaml` + `LOCAL:*` in `extensions.yaml` | Form match (novel vs novella vs graphic novel) |
| hard_no | `hard_nos.yaml` (entries may carry `exceptions`, see below) | Filter — strict or soft, exception-aware |
| condition | *(sparse, not in this directory — structured in extraction schema)* | Location / word-count / protagonist constraints |

> **"Independent" means orthogonal codebooks**, not that profiles or sections are independent of each other. Sections within a profile can cross-reference each other (e.g. *"Genres: anything listed in other sections"*); L3 resolves those references during canonicalization. See `docs/features/16_l3_canonicalization.md` for the section-graph handling.

> **Hard-nos are not strictly binary.** Real profiles include conditional exclusions ("Fantasy except very light"). The canonical hard-no shape is `{tag, severity, exceptions: ConstraintExpr[]}`; the matcher evaluates exceptions before rejecting. Severity (`strict` / `soft`) stays as defined in `hard_nos.yaml`.

## Alias resolution algorithm (runtime)

This applies to **atomic coord tokens** (single-phrase subject / audience / form / hard-no labels). Compound expressions (`"Cozy fantasy (light only)"`, `"RomCom WHERE both leads are POC"`, `"Fantasy except very light"`) are *not* exhausted by alias lookup — they go through the structured L2 (interpretation) and L3 (canonicalization) path described in feature docs 17 and 16.

1. Normalize input: lowercase, strip punctuation, collapse whitespace.
2. Exact lookup in `aliases.yaml` under the relevant facet.
3. Miss → log to `unmapped_terms` review table with `{facet, raw_term, profile_id, canon_version}`. **Do not invent a code.**

For atomic lookup: no fuzzy match, no embedding fallback, no LLM rescue. For compound expressions: split into `coord + qualifier/exception/constraint`, route the coord part through this lookup, store the qualifier as structured metadata. Ambiguity beyond either path is deferred to humans via the review log.

## Extending the canon

> **Runtime auto-extension is forbidden.** New categories appear only through this workflow. See `docs/features/16_l3_canonicalization.md` for why.

1. During the review cycle, inspect `unmapped_terms` grouped by normalized raw term.
2. For each cluster:
   - **Existing code, missing alias** → add to `aliases.yaml`. No version bump unless substantial.
   - **Real gap** → add a `LOCAL:*` entry to `extensions.yaml` **only if**:
     - Appears in ≥5 distinct profiles, AND
     - Cannot be expressed as a combination of existing codes, AND
     - Has broad market recognition (not a single-agent idiosyncrasy).
   - **Noise / idiosyncratic** → dismiss in the log; no canon change.
3. Bump `VERSION` (patch for alias-only, minor for new extension, major for Thema upstream bump).
4. Run `scripts/canon_coverage.py` and commit the updated `coverage_report.md` alongside the canon change.
5. Re-tag affected agent profiles against the new canon version.

## Regenerating Thema files

Only when moving to a new upstream Thema release:

```bash
# Download new JSON into canon/_source/thema_vX.Y_en.json
# Update the SRC path in canon/_source/filter_thema.py
python3 canon/_source/filter_thema.py
```

This rewrites `thema_subjects.yaml`, `thema_audience.yaml`, `thema_form.yaml`. Hand-review the diff before committing — upstream Thema occasionally renames codes.
